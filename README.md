# Тестовое задание: Backend-разработчик

Платформа «Умный реестр подписок» (Backend-ядро и AI-интеграция) для автоматизации учёта,
расчёта дат списаний и контроля статуса подписок пользователя.

REST API для управления обязательствами (`obligations`) и платежами (`payments`).

Стек:

- Java 21
- Spring Boot
- Spring Data JPA / Hibernate
- PostgreSQL 16
- Maven
- JUnit
- Mockito
- Flyway
- Docker Compose
- Swagger OpenAPI

# 3. Почему lazy expiry не применяется к рекуррентным обязательствам

## Правило “ленивого истечения” (Lazy Expiry)

Обязательства со статусом `ACTIVE` и 
датой `next_payment_date` меньше текущей даты автоматически переводятся 
в статус `EXPIRED` при получении списка через `GET /obligations`.

### Исключение: рекуррентные обязательства

Обязательства, у которых `recurrence != null`, исключаются из этого правила.
Даже если их `next_payment_date` находится в прошлом, они остаются в статусе `ACTIVE`.

### Обоснование

Рекуррентные обязательства представляют собой не разовые платежи, 
а **долгосрочные подписки или регулярные платежи**.
Просроченная дата в этом случае не означает завершение обязательства, а лишь указывает на:

- пропущенный платёж
- или ожидающийся цикл оплаты

Поэтому автоматическое завершение таких обязательств было бы некорректным 
с точки зрения бизнес-логики и привело бы к ошибочному закрытию активных подписок.

### Архитектурное решение

Подход lazy expiry применяется на этапе чтения данных (read-time), 
а не через фоновые задачи или планировщики.

Это позволяет:

- обеспечить согласованность данных без асинхронных процессов
- избежать необходимости в background job / scheduler для простой бизнес-логики
- сделать поведение системы предсказуемым в момент запроса

# 4.  Как обрабатываются граничные случаи с датами при рекуррентности

## Логика обработки рекуррентных платежей

При оплате рекуррентного обязательства следующая дата платежа 
рассчитывается от текущего значения `next_payment_date`, 
а не от фактической даты оплаты. 
Это предотвращает смещение графика платежей, если оплата была произведена с задержкой.

Для расчёта дат используются методы 
`LocalDate.plusMonths()` и `LocalDate.plusYears()` из Java, 
которые корректно обрабатывают случаи с концом месяца. 
Например:
- `2026-01-31 + 1 месяц = 2026-02-28`
- `2028-01-31 + 1 месяц = 2028-02-29` (високосный год)
- `2024-02-29 + 1 год = 2025-02-28`

# 5. Какие компромиссы сделал и что бы сделал иначе при наличии большего времени

## Сделано
Lazy expiry

Выбрана модель обновления статуса при чтении данных.

Плюсы:

- простая реализация;
- не требует фонового scheduler;
- меньше инфраструктуры и проще сопровождение.

Минусы:

- статус может оставаться неактуальным до первого запроса;
- дополнительная нагрузка на операции чтения из-за проверки и возможного обновления статусов;
- чтение данных может приводить к изменению состояния системы (побочный эффект GET);
- не подходит для интеграций и уведомлений, которым требуется своевременное изменение статуса;
- при увеличении объема данных и числа пользователей возрастает нагрузка на операции чтения;
- невозможно гарантировать, что все просроченные обязательства будут обновлены к определенному моменту времени.

## Что можно улучшить при большем времени

### 1. Добавить отдельный scheduler для expiry, который ежедневно проверяет просроченные обязательства.

```
@Scheduled
```

### 2. Добавить аудит изменений

Создать историю изменений обязательств.

```
obligation_history

id
obligation_id
action
old_value
new_value
created_at
```

Позволит отслеживать:

- изменение суммы;
- перенос даты оплаты;
- отмену;
- оплату.

### 3. Добавить optimistic locking

Для защиты от конфликтующих изменений:

```
@Version
private Long version;
```

Это поможет избежать ситуации:

```
Request A читает obligation

Request B читает obligation

Request A изменяет

Request B перезаписывает изменения A
```

### 4. Поддержка нескольких валют

Добавить сервис курсов валют.

Например:

```
EUR
USD
RUB
```

И получать общую сумму в выбранной валюте.

### 5. Логирование действий

Кроме аудита БД добавить структурированные логи.

Например:

```
INFO Obligation created

INFO Obligation cancelled

INFO Payment completed
```

### 6. Массовые операции

Например:

```
DELETE /obligations
```

или

```
PATCH /obligations/cancel
```

для нескольких обязательств сразу.

### 7. Увеличить покрытие тестами

Расширить покрытие **unit-тестами** для сервисного слоя, 
добавив больше проверок граничных случаев, обработок ошибок и бизнес-логики.
Дополнительно реализовать интеграционные тесты 
с использованием **Testcontainers** и **MockMvc** для проверки работы REST API, 
взаимодействия с базой данных и корректности выполнения бизнес-сценариев в условиях, 
приближенных к реальным.
